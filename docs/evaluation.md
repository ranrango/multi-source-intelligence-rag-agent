# 评估方案

## 评估目标

这个项目重点评估“多源证据能否支撑一个可追溯研判报告”，而不是只评估回答是否流畅。

## 核心指标

| 指标 | 说明 | 合格标准 |
|---|---|---|
| Evidence Recall | 问题是否召回关键来源证据 | 样例问题至少命中传感器、内部通知、公开信息三类来源 |
| Citation Accuracy | 报告引用是否能对应到真实 chunk | 每条关键结论至少有一个证据编号 |
| Risk Consistency | 风险等级是否符合规则和证据 | 夜间、低空、关键区域、计划外停留同时出现时应提升风险 |
| Uncertainty Quality | 是否说明不确定性 | 报告必须区分事实、推断和待复核项 |
| Trace Completeness | 是否记录 Agent 步骤 | 检索、实体抽取、时间线、风险评估都应写入 trace |

## 手工评测集

建议维护一个 `eval_cases.jsonl`，每条包含：

```json
{
  "question": "北岭港东侧堤坝无人机活动是否异常？",
  "expected_sources": ["sensor", "internal_notice", "social_media"],
  "expected_risk": "高",
  "must_include": ["雷达", "维护", "悬停", "人工复核"]
}
```

## 失败归因

- 召回不足：调整 chunk、关键词、向量检索或重排序。
- 引用错误：限制报告只能使用 evidence 中的内容。
- 风险误判：拆分规则权重，引入人工标注校准。
- 幻觉：增加证据充足性判断和拒答逻辑。

## 上线监控

- 每日请求量、失败率、平均延迟。
- Top-K 证据命中率。
- 人工复核通过率。
- 高风险报告的误报率和漏报率。
- trace 缺失率。
